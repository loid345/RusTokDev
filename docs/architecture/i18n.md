# Архитектура i18n

Этот документ фиксирует i18n-контракт текущего состояния в RusToK: где выбирается
effective locale, как устроен storage-контракт и как UI должен потреблять язык.

## Основной принцип

i18n в RusToK не является локальной договорённостью отдельного модуля или host-а.
Это платформенный контракт.

Следствия:

- effective locale выбирается server/runtime-слоем
- storage и transport используют единый locale-контракт
- module-owned UI packages не вводят собственную locale fallback-chain
- локализованные данные живут в parallel records, а не хаотично в base rows

## Effective locale

Effective locale выбирается host/runtime-слоем на основе:

- tenant locale-policy
- request-контекста
- server-side normalization и validation

UI-package получает уже вычисленный effective locale и не должен сам выбирать его
через query/header/cookie fallback chain.

## Storage-контракт

Текущий целевой storage pattern:

- language-agnostic state живёт в base tables
- локализованные короткие поля живут в `*_translations`
- тяжёлый локализованный контент может жить в `*_bodies`
- locale хранится в нормализованном виде с безопасной шириной `VARCHAR(32)`

Если старый модуль использует inline localized JSON или смешанный storage,
это считается transitional path.

## UI-контракт

Для UI действует единое правило:

- Leptos hosts используют host-provided locale-context
- Next hosts используют runtime/host locale-providers
- module-owned UI packages только потребляют effective locale
- package-local locale negotiation запрещена
- shared support crates для module-owned UI тоже подчиняются этому правилу:
  они не вводят editable locale fields и не inventят свой fallback chain поверх host locale

Это правило одинаково важно для admin, storefront и capability-owned UI-поверхностей.

## Transport-контракт

i18n должен быть согласован на всех слоях:

- request locale selection
- GraphQL и REST payload semantics
- module-owned UI bundles
- email/template rendering
- search/index/read-model projections, если locale влияет на read-side

Transport-слой не должен иметь свой отдельный locale-контракт, расходящийся с
runtime и storage.

## Верификация

Минимальные i18n gates:

- `npm run verify:i18n:ui`
- `npm run verify:i18n:contract`
- host/module-checks, если меняется locale-aware routing или bundle wiring

Если меняется i18n-контракт модуля, нужно обновлять и его local docs.

## Текущий runtime status

- Next hosts берут supported locales из host-owned message loaders и не держат
  отдельный package-local locale registry.
- `apps/next-frontend` использует общий locale-aware middleware matcher без
  hardcoded `/en|/ru` route filter.
- `apps/next-admin` выбирает effective locale на host/runtime уровне:
  query `locale`, cookie `rustok-admin-locale` и request headers нормализуются в `x-rustok-effective-locale`
  в порядке `?locale` → cookie → header → `Accept-Language` → `en`,
  а UI и module-owned packages потребляют результат через `next-intl` / `<html lang>`.
- Leptos hosts используют generated locale enum из текущих locale bundles; добавление
  нового языка начинается с host-owned bundle/генерации, а не с module-local fallback chain.
- Server locale middleware ограничивает effective locale tenant policy из `tenant_locales` через
  bounded TTL cache; counter-метрики `rustok_tenant_locale_cache_hits_total`,
  `rustok_tenant_locale_cache_misses_total`, `rustok_tenant_locale_db_queries_total`,
  `rustok_tenant_locale_cache_invalidations_total` и gauge
  `rustok_tenant_locale_cache_entries` используются для контроля hot-path DB amplification.

## Что не делать

- не вводить module-local locale chain
- не считать UI bundle-контракт отдельным source of truth от runtime locale-policy
- не хранить локализованный бизнес-текст в base rows, если модуль уже идёт по
  parallel-localized-record path
- не разносить locale normalization по нескольким несогласованным слоям
