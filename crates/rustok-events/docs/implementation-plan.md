# План реализации `rustok-events`

## Текущий статус

- Статус: **Phase 1 (extraction in progress)**.
- Кратко: crate уже используется как стабильная точка импорта событийных контрактов,
  но фактические типы всё ещё определяются в `rustok-core` и только реэкспортируются.

## Gap-анализ

### Что уже сделано

- Сформирован выделенный crate `rustok-events`.
- Обеспечена обратная совместимость через реэкспорт `DomainEvent` и `EventEnvelope`.
- Зафиксирована цель постепенного выделения событийных контрактов из `rustok-core`.

### Что отсутствует

- Собственное размещение канонических определений event-контрактов внутри `rustok-events`.
- Формальный policy по versioning/deprecation для событийных payload-изменений.
- Расширенный набор контрактных тестов на совместимость сериализации/десериализации.
- Явные release-gates для breaking/non-breaking изменений событийной схемы.

## Этапы работ

### Этап 1 — Контрактная стабилизация

- Подтвердить текущий public API как единственную рекомендованную точку импорта.
- Добавить документацию по правилам эволюции событийных payload.
- Зафиксировать non-breaking требования к изменению event-полей.

### Этап 2 — Перенос источника истинны

- Перенести канонические типы событий из `rustok-core` в `rustok-events`.
- Сохранить адаптерный слой в `rustok-core` для совместимости на переходный период.
- Провести миграцию зависимостей потребителей на новый источник типов.

### Этап 3 — Production hardening

- Добавить contract/compatibility tests для версионности событий.
- Ввести release checklist для проверки backward compatibility.
- Увязать observability/incident runbook с изменениями в событиях (DLQ/replay/reindex).

## Критерии готовности

- Все внутренние `rustok-*` crates импортируют event-контракты через `rustok-events`.
- Канонические определения `DomainEvent`/`EventEnvelope` находятся в `rustok-events`.
- Для каждого изменения схемы события есть documented migration/deprecation note.
- Контрактные тесты проходят для актуальной и предыдущей поддерживаемой версии payload.

## Метрики верификации

- **Adoption rate:** доля crates, где импорты event-контрактов идут через `rustok-events` (целевое значение: 100%).
- **Compatibility health:** доля успешных contract tests между версиями (целевое значение: 100%).
- **Breaking-change leakage:** число незадокументированных breaking changes в событиях за релиз (целевое значение: 0).
- **Docs freshness:** наличие обновлений в `docs/index.md` и модульной навигации при структурных изменениях (целевое значение: 100% релевантных PR).
